# G3-V3 — fill_risk on honest fills: the decisive merge-maker test

**Date:** 2026-06-12
**Builds on:** `G3-QUEUE-FILL-BACKTEST-2026-06-12.md` (control −$1,130, locked
margin −$106) and `G3-V2-COST-GUARD-2026-06-12.md` (cost-guard fixed locked
margin −$106→+$838, 56→0 negative-margin windows; paired(≤10) bucket profitable
in every config; **100% of the remaining loss is directional residual from
unpaired inventory**).
**Same data, no new fetch:** PMXT v2 hourly parquet 2026-06-10T14–T20 UTC,
btc/eth-updown-5m, 162 windows, real Gamma 0/1 settles, real Binance 1m fair
value, back-of-queue queue-fill (250 ms ack, prorata cancel, 2 s re-quote).

This round tests the only untested lever that targets unpaired-inventory
**formation**: `fill_risk` (TradingBot2 `rails/fill_risk.py` +
`merge_maker.apply_fill_risk_overlay`), ported verbatim with its **shipped
default thresholds** (no tuning on this window — tuning here would invalidate
the test). It scores directional tape pressure on each resting BUY using the
real replayed trade tape + book touch → LOW/MOD/ELEVATED/HIGH → ELEVATED widens
N ticks, HIGH pulls the side, BEFORE the queue-fill step.

## VERDICT

**fill_risk IS the lever.** Acting on honest, back-of-queue fills it does
exactly what it was built to do and the effect is large and monotone:

| transition | unpaired shares | residual settle | TOTAL | active windows |
|---|---|---|---|---|
| (b) guard → (e) guard+fr | **5,926 → 1,548 (−74%)** | **−$1,821 → −$161** | **−$902 → −$79** | 146 → 75 |
| (c) guard+cap10 → (f) +fr | **2,856 → 1,242 (−57%)** | **−$920 → −$142** | **−$528 → −$60** | 146 → 75 |

It cut unpaired inventory by 57–74%, collapsed the residual ~90% toward zero,
and improved total PnL by **+$824** (b→e) and **+$468** (c→f). e≪b and f≪c on
every metric the test pre-specified. The merge-maker mechanism **does survive
honest fills** in the sense that the catastrophic −$1,130/−$902 bleed is
removed: with the cost-guard AND fill_risk acting, the lane lands at
**−$79 / −$60 — essentially breakeven on this window**, median per-window PnL
**$0.00**, with positive locked margin and the paired bucket still profitable.

**But read the honest caveat before declaring victory:** it reaches near-zero by
**trading far less** — active windows 146 → 75, completion 76% → 54%, filled
shares 25,095 → 3,384 (b→e). fill_risk's correct response to a venue where
resting BUYs are adversely selected is to *get out of the way* (13,479 pulls +
9,284 widens in config e). So the surviving strategy is a **much smaller, far
more selective maker** that is **marginally negative, not clearly profitable,**
on this single window. The lane is **salvageable, not yet proven** — it is no
longer structurally broken, and there is a real, mechanism-level reason it
improved (less adverse inventory), which is the bar for "keep developing"
rather than "reassess Lane A." It does NOT clear the bar for "deploy and size
in" — that needs a forward track showing the selective maker is net-positive
out of sample, not merely de-risked to ≈0 in-sample.

## What fill_risk actually did

- Real adverse-tape detection on the replayed prints: in config e it **pulled a
  side 13,479 times and widened 9,284 times** across the run — i.e. on most
  decision ticks at least one side was at ELEVATED/HIGH risk. This is consistent
  with G3's core finding that these books adversely select resting BUYs; the
  overlay is correctly reading a genuinely hostile tape, not over-firing on
  noise (it's quiet enough that completion is still 54–58%, not 0%).
- The residual *win-rate* barely changed (b 20/137=15% → e 23/60=38%; still a
  losing bet when it happens), but the residual *exposure* shrank ~74%, so the
  dollar bleed collapsed. fill_risk works by **avoiding the bet**, not by making
  the unpaired inventory win — which is the right and only honest mechanism.

## CALIBRATION / HONESTY

- **fill_risk used shipped defaults, untouched** (short_s 30 / long_s 3600,
  count/size denoms 2/1000 & 20/5000, floor 0.1, prox_mult 0.9, base_widen 2,
  classify cutoffs .25/.50/.75 — all copied from the Python reference into
  `src/lib/strategies/fill-risk.ts`). NOTHING was tuned on this 7-hour window.
  The marginal effect (e vs b, f vs c) is therefore an honest read of the
  shipped model, not an in-sample fit.
- **The cost-guard (V2) is still the only piece I'd call deploy-ready as-is** —
  it enforces an invariant. fill_risk is a *parameterized behavior*; its
  default thresholds happened to land near-breakeven here, but "near-breakeven
  in-sample by trading 50% less" is a hypothesis to forward-test, not a banked
  edge. cap-10 (configs c/f) remains an in-sample parameter choice, also not
  banked.
- **Effective N is still ~2 symbol-days.** 162 windows but 7 contiguous UTC
  hours of ONE day, one vol regime, BTC and ETH sharing every hour and every
  Binance move; windows within a symbol-hour share book state. The btc/eth
  split is ≈2 independent symbol-days. Critically, after fill_risk only **75
  windows have ANY fills** — the selective maker's effective sample is *smaller
  still*, ~one symbol-day per arm. A −$79 result on ~75 sparse, correlated
  windows is **not** distinguishable from breakeven, and is one regime. This
  cannot carry a deploy decision.
- **Same fill-model limits as G3/V2 carry over:** MBP≠MBO (prorata neutral;
  verdict was bracket-invariant in G3); per-token `last_trade_price` tape
  (cross-book mint fills may be under-counted → completion is a lower bound,
  which would *help*); flat 250 ms ack; no rate limits / $1 rebate threshold /
  LP rewards; Binance 1m spot lags the live WS; strike = Binance 1m open vs
  Polymarket's own oracle. The fill_risk tape itself is the same
  `last_trade_price` stream — if cross-book prints are missing, fill_risk sees
  *less* adverse pressure than reality, so its real-world effect could be even
  stronger (more pulls), further shrinking volume.

## Recommendation

**Keep Lane A alive, advance to forward paper — do NOT reassess/kill it, and do
NOT size in.** The mechanism is no longer structurally broken: cost-guard
(deploy-ready) + fill_risk-acting (shipped defaults) removes the −$1,130 → ≈$0
bleed through a real, mechanism-level cause (74% less adverse inventory). That
clears the "is the mechanism salvageable on honest fills?" bar this test was set
to decide: **yes, salvageable.** The next gate is the only one that can settle
it — a **forward paper track with independent resolution** running cost-guard +
fill_risk-acting, judged on whether the now-selective maker is net-positive out
of sample. Until that holds, treat −$79/−$60 as "de-risked to breakeven
in-sample," not as profit.

What NOT to do next: tune fill_risk thresholds or the cap on this window to push
−$79 below zero. That is precisely the overfitting trap; the mild(10–30) residual
is the visible temptation and must be left for forward data.
